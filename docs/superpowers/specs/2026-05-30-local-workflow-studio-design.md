# Nexus Design

Date: 2026-05-30

## Product Thesis

Nexus is a macOS app for AI-generated automations that run locally. The user describes a workflow in plain English, reviews the generated plan, dry-runs it, approves any risky permissions or scripts, and saves it to a local runner.

Tagline:

> Generated by AI. Run locally on your Mac.

The product is not positioned as "local AI" because the planning model can be cloud-based. The execution, logs, approvals, and workflow state are local by default.

## First Users

The first audience is students, builders, and power users who have repetitive local chores but do not want to hand-write scripts.

High-value early workflows:

- Organize screenshots and visual files.
- Sort Downloads, PDFs, class files, and project assets.
- Open study or dev routines.
- Record and replay repeated browser tasks.
- Use guarded UI control when no clean automation API exists.

## MVP Target

The first build is a hackathon demo with a serious-MVP spine:

- Polished chat-first product shell.
- Real screenshot cleaner.
- Real controlled-browser recorder/replay demo.
- Real warnings, dry run, run logs, and trust approval.
- Raw generated scripts allowed behind explicit approval.
- macOS Accessibility fallback requests permission only when an automation needs UI control.
- Advanced block/script editor exists, even if not every future block type is fully implemented.

## Core Flow

1. User describes an automation in chat.
2. AI planner asks follow-up questions when ambiguity affects the result, especially for risky actions.
3. AI produces workflow JSON plus an editable review.
4. Review screen shows warnings, affected files/apps/actions, scripts, permissions, and undo availability.
5. User dry-runs the workflow.
6. User approves any raw script or expanded permission set.
7. Local runner executes the workflow and records a run log.

## Architecture

```text
Chat prompt
  -> cloud AI planner
  -> workflow JSON
  -> review UI / advanced editor
  -> local runner
  -> macOS adapters
```

Core components:

- Chat UI: primary automation creation surface.
- Workflow schema: restricted, reviewable JSON representation.
- Advanced editor: block and script inspection/editing.
- Trust engine: warnings, approvals, permission diffs, re-approval rules.
- Dry-run engine: predicts exact impact before execution.
- Local runner: executes workflows on the user's Mac.
- Log store: records automation runs locally.
- Adapter layer: files, shell, AppleScript, controlled browser, Accessibility, notifications.

## Workflow Schema

Workflows should be represented as structured JSON before execution. Raw scripts are allowed, but they are wrapped as explicit action blocks with warnings and approval requirements.

Example:

```json
{
  "name": "Screenshot Cleaner",
  "trigger": { "type": "manual" },
  "actions": [
    {
      "type": "find_files",
      "folder": "~/Desktop",
      "match": "Screen Shot*.png"
    },
    {
      "type": "move_files",
      "destination": "~/Pictures/Screenshots/{yyyy-mm-dd}"
    },
    {
      "type": "notify",
      "message": "Screenshots organized"
    }
  ]
}
```

## Trust Model

Warnings use simple language instead of legal/security jargon.

Examples:

- This can delete files.
- This can read files in Downloads.
- This can type into other apps.
- This can click buttons you may not expect.
- This can send data to a website.
- Undo may not be possible.

Raw scripts can run after one explicit trust approval. Re-approval is required when:

- The script changes.
- AI edits the automation.
- Permissions expand.
- Trigger changes to scheduled or background execution.
- UI-control ability is added.
- Network access appears.

Accessibility fallback never runs silently. It must show the app it will control, the actions it will take, a stop hotkey, permission warnings, visible running state, and a run log.

## AI And Privacy

AI planning can use a cloud model. Local execution stays on the Mac.

Default privacy rules:

- User prompt and workflow metadata can go to the AI provider.
- File contents are not uploaded by default.
- Screenshots are not uploaded by default.
- UI recordings are not uploaded by default unless the user explicitly approves.
- Logs and runner state stay local by default.

## Dry Run

Dry run is the primary trust surface.

For file workflows, dry run shows:

- Files that will be read.
- Files that will move, rename, archive, or delete.
- Destination paths.
- Whether undo is available.

For browser/UI workflows, dry run shows:

- Target app or browser.
- Each click/type/navigation step.
- Whether targeting uses DOM selectors, accessibility elements, image matching, or coordinates.
- Stop hotkey and permission state.

Warnings are grouped at the top of the dry-run screen.

## Logging

Default logging is per-run status:

- Automation name.
- Timestamp.
- Trigger source.
- Success or failure.
- Actions attempted.
- User-facing error message.

User-selectable options:

- Detailed step logs.
- Full file path detail.
- Screenshots for UI automation debugging.
- Log retention period.
- Clear logs anytime.

## First Real Demos

### Screenshot Cleaner

User prompt:

> Every night, organize my Desktop screenshots into dated folders.

Functional scope:

- Manual run for demo.
- Finds screenshots on Desktop.
- Moves them into dated folders.
- Shows exact files in dry run.
- Writes run log.
- Supports undo when possible.

### Controlled Browser Automation

User prompt:

> Record this browser task and let me replay it later.

Functional scope:

- App launches a controlled browser session.
- User records clicks and typing.
- Recorder saves stable web steps where possible.
- Review screen previews steps.
- Runner replays the task.
- Logs success/failure.

### Accessibility Fallback

Functional scope:

- Available when a workflow requires computer control beyond structured browser automation.
- Requests macOS Accessibility permission only at the moment it is needed.
- Shows a plain-language reason such as: "This automation needs permission to click and type in Safari."
- Requires explicit approval before running.

## UX Surfaces

- Chat screen: primary place to describe automations.
- Review screen: warnings, dry run, permissions, scripts, and actions.
- Advanced editor: blocks, raw scripts, triggers, logs, and undo behavior.
- Automation dashboard: saved workflows and latest run status.
- Logs screen: run history and debugging options.
- Settings: AI provider, privacy defaults, log retention, permissions.

## Non-Goals For First Build

- Browser extension recorder.
- Full Shortcuts export.
- Fully local model inference.
- Full background scheduling if time is tight.
- Broad native app automation coverage.
- Marketplace or sharing workflows.
- Cloud sync.

## Risks

- Raw scripts can damage files if warnings and approvals are weak.
- UI automation can be brittle when screens change.
- Accessibility permissions can scare users if requested too early.
- "Local" wording can mislead users if cloud AI planning is involved.
- Controlled browser demos may not use the user's real logged-in sessions.

## Verification Plan

Minimum proof for the first build:

- Screenshot cleaner dry run shows exact files before moving.
- Screenshot cleaner actually moves files in a test folder.
- Undo works for moved files or clearly reports when undo is unavailable.
- Controlled browser recorder can record and replay a simple task.
- Raw script approval stores a trusted version and asks again after changes.
- Accessibility fallback requests permission only when needed.
- Run logs appear after success and failure.
- No file contents, screenshots, or UI recordings are uploaded by default.

## Spec Self-Review

- No unfinished requirements remain.
- MVP scope is limited to a hackathon demo with real trust and runner foundations.
- The privacy claim distinguishes AI generation from local execution.
- Autoclicking is a fallback, not the primary automation primitive.
- Risky actions require follow-up questions, warnings, dry run, and approval.
